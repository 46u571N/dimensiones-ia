Spartan IT · arquitectura · dimensiones · v1.0

Tu producto madura
por dimensiones,
no en bloque.

Auth en fase 03, RAG en fase 01, billing en fase 02. Es completamente normal. Esta es una guía para entender qué cambia en cada dimensión por separado, y cuándo conviene madurar cada una.

tesis · pensar la madurez "del producto" como un número único (fase 1, 2, 3) lleva a errores. el producto es un vector con muchas componentes — cada dimensión avanza a su propio ritmo según el cliente, la regulación y la presión del negocio. acá desarmamos las seis críticas para spartan, y catalogamos las ocho restantes.
no confundir · las "fases 00-03" de cada dimensión acá son específicas de esa dimensión (datos, auth, rag, etc). la app de las cuatro fases usa los mismos nombres pero refiere a fases del producto entero (mcp+claude desktop → agente propio → saas multi-tenant). son modelos complementarios, no el mismo.
Dimensión 01

Datos y conexión a base

Cómo el agente accede a los datos del cliente. Probablemente la dimensión más conflictiva de Spartan, porque ningún cliente serio te va a dar acceso directo a su base productiva — y eso obliga a diseñar capas intermedias desde el principio.

FASE 00 Explorar
No hay conexión a datos

Si necesitás datos reales, los pegás a mano en claude.ai. Toda la conexión vive en tu cabeza y tu portapapeles.

FASE 01 Conectar
Conexión directa a la base del cliente

Tu MCP server con credenciales hardcoded en variables de entorno. Conexión única, sin pool, sin retry sofisticado.

  • Postgres / MySQL / SQL Server según lo que tenga el cliente
  • Credenciales en .env del MCP server
  • Una sola DB, vos sos el único usuario
  • El cliente probablemente te dio un usuario SQL de solo lectura
FASE 02 Operar
Vistas, réplicas o API que el cliente expone

El cliente casi nunca te abre la DB completa. Te entrega vistas filtradas, una réplica en tu GCP, o una API REST que ellos exponen y vos consumís.

  • Pool de conexiones (no abrir una por request)
  • Secretos en GCP Secret Manager, no en código
  • Capa de abstracción (repository pattern) entre la tool y la DB
  • Retry con backoff exponencial
  • Read replica si el volumen es alto
FASE 03 Escalar
Una conexión por tenant, aislamiento estricto

Cada cliente tiene su propia conexión, su propio pool, sus propios secretos. El código resuelve qué conexión usar según el tenant activo en cada request.

  • TenantContext middleware en Laravel
  • Connection pool por cliente, configurado en runtime
  • Secretos por tenant en Vault o Secret Manager con namespaces
  • Row-level security donde sea posible
  • Backup y DR diferenciado por SLA del tenant

› aspectos que cambian fase a fase

Aspecto Fase 01 Fase 02 Fase 03
Dónde viven las credenciales .env en el host del MCP GCP Secret Manager Vault con namespace por tenant
Modelo de acceso Usuario SQL único, read-only Vistas filtradas, réplica, o API REST Conexión resuelta dinámicamente por tenant
Pool de conexiones Una conexión, sin pool Pool fijo (Laravel default) Pool por tenant con límites configurables
Manejo de errores Excepción al usuario Retry con backoff, fallback genérico Circuit breaker, SLA degradado configurable
Aislamiento de datos No aplica (vos sos el único) No aplica (un cliente) Crítico — fuga entre clientes es incidente grave

Próximos pasos al saltar de fase

01 → 02
Negociar la conexión con el cliente antes de prometer features. La cerealera probablemente no te abre la DB. Pedile vistas SQL específicas con los campos que el agente necesita, o que te expongan endpoints REST. Definí el contrato de datos en escrito antes de codear.
02 → 03
Refactorizar a connection pool por tenant. Si no hiciste el campo client_slug desde el día uno, este salto es doloroso. Implementá un middleware que resuelva el tenant del request y lo inyecte en el contexto. Las tools pasan a recibirlo como dependency.
› dónde está spartan hoy

Cerealera: directo a fase 02 por restricción del cliente

Ya validaste con el cliente que no te abren la DB — te van a dar vistas parciales. Eso significa que para la cerealera se saltean fase 01 y arrancan directo en fase 02. Vale la pena empezar diseñando el repository pattern y los secretos en Secret Manager desde el primer día, para no refactorizar después.

Dimensión 02

Auth y observabilidad

Quién puede ver qué, quién hizo qué, y cuándo te enterás de que algo falla. Para clientes regulados (lab, clínica, salud) esta dimensión no es opcional — es la diferencia entre tener producto vendible y no.

FASE 00 Explorar
Vos mismo, sin nada

Te logueás a claude.ai con tu cuenta personal. No hay nada que auditar excepto tu propio historial.

FASE 01 Conectar
Claude Desktop maneja la sesión

Vos y Alberto se conectan con sus cuentas de Anthropic. El MCP server confía en que quien se conecta es legítimo.

  • Sin auth en el MCP server (no es necesario, corre local o en tu red)
  • Logs del MCP a archivo, mirás a mano
  • Sin métricas, sin alertas
FASE 02 Operar
SSO + roles + logs estructurados

Cada usuario del cliente entra con su cuenta. Roles básicos para limitar qué puede hacer. Logs JSON que vos podés grepear y agregar a un dashboard.

  • Auth con Laravel Sanctum, Google SSO, o Auth0
  • Roles: admin / user / viewer
  • Logs estructurados (JSON) con request_id
  • Dashboard de uso (Langfuse o propio)
  • Alertas básicas (mail si error rate > 5%)
FASE 03 Escalar
OIDC por tenant + tracing distribuido

Cada cliente puede traer su propio Identity Provider. Permisos finos por recurso. Tracing distribuido para debuggear flujos cross-service.

  • OIDC / SAML — el cliente trae su Azure AD, Okta, etc.
  • Scopes finos (puede ver saldos pero no facturas)
  • Row-level security donde es posible
  • OpenTelemetry para tracing
  • SLA por cliente, alertas dirigidas según severidad y tenant
  • Retención de logs diferenciada por compliance del cliente

› aspectos que cambian fase a fase

AspectoFase 01Fase 02Fase 03
Identidad del usuario Cuenta Anthropic SSO o user/pass propio OIDC del tenant
Granularidad de permisos Todo o nada 3 roles fijos Scopes por recurso, custom por tenant
Log de auditoría Archivo plano del MCP Tabla audit_log con quién/qué/cuándo Inmutable, append-only, retention configurable
Métricas operativas A ojo Dashboard con latencia, errores, costo Por tenant, con percentiles y comparativas

Próximos pasos al saltar de fase

01 → 02
Definir el modelo de auth antes de la UI. Es más fácil agregar auth al backend que retrofitearlo después. Decidir temprano: SSO con Google si el cliente es PyME, Auth0 si tiene IT serio. Diseñar la tabla audit_log el día uno.
02 → 03
OIDC y tracing en paralelo. Ambos son trabajo grande y se hacen juntos porque comparten infraestructura (request_id viaja con el token, los traces los hereda). OpenTelemetry tiene curva de aprendizaje pero es el estándar.
› dónde está spartan hoy

Lab: fase 01 con plan de salto a fase 02 ya en marcha

Hoy el lab corre con MCP + Claude Desktop, sin auth formal. Eso es OK para la demo pero para venderle al lab como producto, fase 02 con auth y audit_log es no negociable — están regulados por ANMAT y necesitan trazabilidad. El plan razonable: SSO con Google Workspace (que ya usan) + tabla de auditoría desde el primer release.

Dimensión 03

RAG y embeddings

Cuando el cliente tiene conocimiento que no vive en bases SQL — manuales, protocolos, historias clínicas, normativas — RAG es la herramienta. Ya tenés inversión técnica (BGE-M3, pgvector) — esta dimensión es cómo evoluciona ese stack.

FASE 00 Explorar
Sin RAG, todo en el prompt

Pegás el documento en claude.ai y preguntás. Funciona para validar si tiene sentido, no escala.

FASE 01 Conectar
Probablemente sin RAG todavía

El MCP del lab hoy resuelve con queries SQL determinísticas, no necesita retrieval. RAG aparece cuando hay documentos no estructurados que el agente necesita consultar.

  • Si hay PDFs / manuales, los embebés a mano y pegás chunks relevantes en el prompt
  • Para queries puramente sobre datos estructurados, no aplica
FASE 02 Operar
pgvector + BGE-M3, retrieval básico

Stack que ya tenés para el lab. Embebés los documentos del cliente, retrieval por similitud coseno, top-k inyectado en el contexto.

  • BGE-M3 como modelo de embeddings (multilingüe, 8K context)
  • pgvector para almacenamiento y búsqueda
  • Chunking de 500 tokens con overlap de 50
  • Top-k entre 4 y 8 con threshold de similitud
  • Re-indexar cuando el cliente actualiza documentos
FASE 03 Escalar
Índices por tenant, re-ranking, evals

Cada cliente con su propio índice. Hybrid search (dense + sparse). Re-ranker liviano para los top-k. Evals continuas para medir si la calidad del retrieval baja.

  • Schema multi-tenant en pgvector con namespace por cliente
  • Búsqueda híbrida: BGE-M3 + BM25
  • Re-ranking con BGE-reranker-base sobre los top-20
  • Cache de embeddings para queries frecuentes
  • Refresh incremental, no full re-index
  • Golden dataset con recall@5 y MRR por cliente

› aspectos que cambian fase a fase

AspectoFase 01Fase 02Fase 03
Estrategia de retrieval A mano o ninguna Top-k denso con threshold Híbrido + re-ranking
Modelo de embeddings Variable o no aplica BGE-M3 (mismo para todos) Posible fine-tuning por dominio
Almacenamiento Archivos sueltos pgvector compartido pgvector con namespace por tenant
Chunking Improvisado Tamaño fijo con overlap Adaptativo por tipo de documento
Actualización del corpus Manual al cambiar algo Script de re-indexado periódico Incremental, trigger por evento
Medición de calidad "Parece que anda" Golden dataset de 30-50 queries Evals continuas, alertas por degradación

Próximos pasos al saltar de fase

01 → 02
Construir el corpus inicial. Pedile al cliente los 50 documentos más consultados. Embeberlos con BGE-M3, indexar en pgvector. Probar retrieval contra preguntas reales antes de exponer al usuario. Definir el threshold con datos, no a ojo.
02 → 03
Aislamiento + re-ranking. Antes de aceptar el segundo cliente, separar índices por tenant (namespace en pgvector). Agregar re-ranker para los casos donde top-k denso trae ruido. Implementar golden dataset por cliente como gate de release.
› dónde está spartan hoy

Stack listo, faltan corpus por cliente

Tenés BGE-M3 + pgvector probado en el lab. La infraestructura técnica está en fase 02. Lo que falta es el corpus específico por cliente — protocolos del lab, manuales de la cerealera, contratos. El cuello de botella no es técnico, es el acceso a los documentos del cliente.

Dimensión 04

Cumplimiento normativo y privacidad

Para tu mercado (lab, clínica, agro, municipios) esta dimensión es el diferencial comercial real. Los competidores que no piensan en regulación quedan afuera de las licitaciones serias. Ya tenés un pie adelantado con la capa de anonimización de Tomás.

FASE 00 Explorar
No aplica formalmente

Si estás trabajando con datos hipotéticos o anonimizados, no hay nada que cumplir. Pero ojo: si pegás datos reales del cliente en claude.ai, ya estás transfiriendo a un tercero — eso puede generar obligaciones bajo la normativa local, conviene revisarlo con un abogado.

FASE 01 Conectar
Vos sos el responsable de hecho

Los datos del cliente pasan por tu MCP server, que vive en tu GCP. Estás actuando como procesador de datos sin haber firmado un acuerdo formal.

  • Ley 25.326 puede aplicar aunque no haya producto formal — revisar con asesoría legal
  • Si son datos de salud, ANMAT Disposición 64/2025 puede generar obligaciones regulatorias adicionales
  • Recomendado: convenio de confidencialidad con el cliente desde el inicio
FASE 02 Operar
Anonimización, consentimiento, retención

Capa formal entre los datos del cliente y el LLM. Política de retención escrita. Mecanismo para que el usuario pida eliminación de sus datos.

  • Anonimización antes de enviar al LLM (ya tenés capa de Tomás)
  • Consentimiento informado del usuario final
  • Retention policy: cuánto guardás los logs, prompts, respuestas
  • Derecho al olvido implementado (borrado real, no soft delete)
  • DPA (Data Processing Agreement) con el cliente
FASE 03 Escalar
Certificaciones formales, residencia de datos

ISO 27001 / HIPAA / SOC2 según el mercado. Datos del cliente almacenados en la región que su regulación requiere. Auditorías externas periódicas.

  • ISO 27001 para certificación general
  • HIPAA si exportás a US o atendés filiales norteamericanas
  • LGPD si Brasil entra al mix
  • Residencia de datos por región (Argentina, Brasil, US)
  • Auditoría externa anual + penetration testing
  • Cláusula contractual de SLA de seguridad

› aspectos que cambian fase a fase

AspectoFase 01Fase 02Fase 03
Marco contractual NDA informal o nada DPA + términos de servicio DPA + SLA seguridad + cláusulas compliance
Anonimización Ninguna o manual Capa automática antes del LLM Tokenización reversible por tenant
Retención de datos Indefinida, descuidada Política escrita, ~90 días default Configurable por tenant según regulación local
Derecho al olvido No implementado Endpoint de borrado del usuario Workflow auditable con SLA de cumplimiento
Certificaciones Ninguna Compliance autodeclarado ISO 27001 + sectoriales (HIPAA, etc.)
Residencia de datos GCP región default GCP sudamerica-este si el cliente lo pide Por tenant, según regulación de su jurisdicción

Próximos pasos al saltar de fase

01 → 02
DPA firmado + capa de anonimización en producción. Antes del primer release a un usuario externo, asesoría legal (no Claude ni un chatbot) revisa el DPA. La capa de anonimización de Tomás pasa de prototipo a feature versionada con tests. Definir retention policy por escrito.
02 → 03
Camino largo a certificación. ISO 27001 tarda 6-12 meses con un consultor. No empezarlo antes de tener flujo de caja predecible. Para clientes que la piden como gate, mientras tanto, presentar evidencias de controles equivalentes.
› dónde está spartan hoy

Capa técnica fase 02, marco legal pendiente

Tenés la capa de anonimización funcionando técnicamente — eso es 50% del trabajo. Lo que todavía no está: DPA modelo para clientes, política de retención escrita, consentimiento del usuario final. Para venderle al lab en producción, esto se hace antes del primer release real. Un abogado especialista en datos personales en LATAM cuesta menos que perder una licitación por no tener el papel.

Dimensión 05

Modelo comercial y pricing

Cómo cobrás determina si el producto tiene margen sostenible o si vas a perder plata escalando. Para una empresa que vende IA, esta dimensión define el techo del negocio. Subdesarrollada en la versión previa de este documento, ahora con detalle propio.

FASE 00 Explorar
No cobrás todavía

Estás validando, no vendiendo. La conversación de precios no aplica.

FASE 01 Conectar
Cobrás por horas o setup

Modelo de consultora: cobrás el armado del MCP server, capacitación al equipo, soporte por hora. El cliente paga su propia suscripción de Claude.

  • Factura única por proyecto + horas de mantenimiento
  • No hay riesgo de pérdida por uso de tokens (vos no pagás el LLM)
  • Pero el ingreso es no recurrente
  • Difícil escalar: cada cliente nuevo = más horas humanas
FASE 02 Operar
Fee mensual fijo, vos pagás el API

Cobrás una mensualidad al cliente que cubre uso esperado + margen. Vos absorbés el riesgo del costo de tokens. Si el uso explota, perdés plata; si está bien estimado, ganás.

  • Ej: USD 800/mes flat para "hasta 3000 consultas mensuales"
  • Necesitás medición fina (ver app de economía de tokens)
  • Overages: por consulta arriba del límite, USD 0.10
  • Reset mensual o trimestral según contrato
FASE 03 Escalar
Tiered SaaS con cuotas y overages

Planes Starter / Pro / Enterprise con cuotas claras, overage por uso, descuentos por volumen anual. Self-service de upgrade. El cliente entiende qué paga y por qué.

  • Starter USD 300/mes: 1000 consultas, 1 usuario admin
  • Pro USD 1500/mes: 8000 consultas, 5 usuarios, SLA 99.5%
  • Enterprise: custom, SLA 99.9%, residencia de datos, soporte 24/7
  • Billing automatizado (Stripe, MercadoPago)
  • Dashboard para que el cliente vea su consumo en tiempo real

› aspectos que cambian fase a fase

AspectoFase 01Fase 02Fase 03
Modelo de ingreso Una vez + horas Recurrente fijo Tiered + overages
Quién paga el LLM El cliente directamente a Anthropic Vos, con margen incluido en el fee Vos, costo por consulta calculado y exposto
Riesgo financiero del prestador Bajo (no pagás tokens) Alto (uso impredecible te puede romper el margen) Controlado (cuotas duras, overages pagan exceso)
Predictibilidad para el cliente Baja (factura variable) Alta (sabe lo que paga cada mes) Alta + flexibilidad de cambio de plan
Esfuerzo de cobranza Manual, factura tradicional Recurrente manual Automatizado (Stripe / pasarela)
Métricas de salud del negocio Facturación total MRR + margen bruto por cliente MRR, churn, NRR, LTV, CAC payback

Próximos pasos al saltar de fase

01 → 02
Calcular el costo real antes de poner precio. Tomá una semana de uso típico del cliente (con la app de economía de tokens) y proyectá costo mensual. Sumá margen 3-4x para cubrir picos de uso, soporte y desarrollo. Nunca cobres "lo que paga el cliente" sin haberlo medido. Pricing por debajo del costo es la forma más rápida de matar la empresa.
02 → 03
Diseñar los planes en base a clientes reales, no en base a competencia. Mirá tus 2-3 clientes actuales: cuántas consultas hacen, qué features usan, qué valor le sacan. De ahí salen los tres planes. Implementar Stripe + dashboard del cliente lleva ~3 semanas si nunca lo hiciste.
› dónde está spartan hoy

Fase 01 con tensión hacia fase 02

Hoy facturás como consultora: setup + horas. Eso funcionó para arrancar pero limita el techo del negocio. El próximo paso natural es pasar a fee mensual fijo con la cerealera o el lab, ahora que ya tenés baseline de uso real. Para hacerlo bien, medir 4 semanas de uso con la app de economía de tokens, calcular costo, sumar 3-4x de margen, y proponer el fee al cliente como "previsibilidad para vos, recurrencia para nosotros".

Dimensión 06

Testing y evals

Cómo medís si el agente responde bien. Para sistemas no determinísticos como los LLM, "parece que anda" no es evidencia válida — y mucho menos para un cliente regulado. Esta dimensión define la diferencia entre un prototipo frágil y un producto vendible.

FASE 00 Explorar
Pruebo y veo

Hacés 5 consultas a mano y juzgás vos mismo si la respuesta tiene sentido. Suficiente para validar la idea.

FASE 01 Conectar
Test manuales del equipo

Alberto prueba consultas reales del lab y anota cuáles fallan. Sin proceso formal, pero hay observación.

  • Documento de "casos que fallaron" en Notion o similar
  • Revisión semanal del documento
  • Sin métricas, sin baseline, sin regresión
  • Suficiente para uso interno, insuficiente para producción
FASE 02 Operar
Golden dataset + evals en CI

Conjunto fijo de 50-100 queries con respuestas esperadas. Cada cambio de prompt, modelo o tool corre el set automáticamente. Si baja la calidad, no se mergea.

  • Golden dataset en formato JSONL versionado en git
  • Evaluator: comparación textual + LLM-as-judge para casos abiertos
  • Métricas: accuracy, recall@k para RAG, tasa de tool calls correctos
  • Ejecución en CI antes de cada deploy
  • Threshold de calidad como gate (ej. accuracy > 85%)
FASE 03 Escalar
Evals continuas en producción + por tenant

Sample de tráfico real evaluado en background. Golden dataset por cliente para reflejar sus casos específicos. Alertas si la calidad drift en producción.

  • Sample del 1-5% del tráfico real con eval async
  • Golden dataset distinto por tenant (cerealera ≠ lab)
  • A/B testing controlado entre versiones de prompt
  • Dashboard de calidad por cliente con alertas por degradación
  • Adversarial testing (red teaming) periódico
  • Para clientes regulados: trazabilidad de qué versión respondió cada query

› aspectos que cambian fase a fase

AspectoFase 01Fase 02Fase 03
Volumen de casos de prueba 10-20 anecdóticos 50-100 estructurados 200+, por tenant, vivos
Frecuencia de evaluación Cuando alguien se acuerda Cada PR / deploy en CI Continua sobre tráfico real
Tipo de juez Humano (Alberto) Comparación textual + LLM-as-judge Multi-judge (LLM + reglas + humano spot-check)
Gate de release "A mí me parece OK" Threshold automático en CI Multi-métrica + signoff manual para releases mayores
Detección de regresión Cliente avisa CI bloquea antes del deploy Alertas en producción por drift

Próximos pasos al saltar de fase

01 → 02
Construir el golden dataset con casos reales. Tomá las 50 consultas más frecuentes del cliente, anotá cuál es la respuesta correcta para cada una, guardalo en JSONL versionado. Implementar el evaluator: comparación textual para casos cerrados, LLM-as-judge (con Claude o el modelo más barato) para casos abiertos. Configurar como step de CI antes del deploy.
02 → 03
Evals continuas + per-tenant. Implementar sampling del tráfico real (1-5%) con eval async usando workers en background. Por cada cliente, un golden dataset propio que refleje sus casos. Definir métricas de drift y alertas. Para clientes regulados (lab), agregar inmutabilidad criptográfica de los logs de eval.
› dónde está spartan hoy

Fase 01 — alto riesgo si el lab pasa a producción

Hoy las pruebas son manuales y depende del ojo de Alberto. Para venderle al lab en producción, fase 02 es obligatorio. Sin evals automáticas, cualquier cambio de prompt puede romper silenciosamente un caso crítico. Y bajo ANMAT, "no nos dimos cuenta" no es defensa válida. Plan razonable: golden dataset de 50 casos del lab + evaluator simple + gate en CI antes del primer release real.

No se aborda todo a la vez

Cuando llegue un cliente nuevo, mirá las 14 dimensiones y preguntate en qué fase razonable está cada una para ese cliente específico. La cerealera probablemente necesite fase 02 en datos/auth/observabilidad, pero quizás fase 01 alcance en RAG (queries SQL determinísticas). El lab necesita fase 02 mínimo en compliance, pero puede arrancar con fase 01 en pricing. Cada cliente tiene su propio vector de madurez. Sin un mapa entero, es fácil sobre-construir en una dimensión y olvidar otra crítica.

› dependencias entre dimensiones

Qué pide qué, qué habilita qué

Las dimensiones no son independientes. Algunas requieren que otras estén maduras primero; otras habilitan que otras puedan avanzar. Saltar este orden lleva a refactorizar después con más dolor.

03Observabilidad y logs
requiere ← Auth (02) · sin identidad estable, los logs no son trazables a usuarios
habilita → Evals (11) · sin métricas en producción no podés evaluar continuamente
habilita → Pricing (07) · sin medir uso real no podés cobrar por consumo
05RAG y embeddings
requiere ← Datos (01) · sin acceso a documentos del cliente no hay nada que indexar
requiere ← Datos (01) fase 03 · RAG multi-tenant exige aislamiento de datos primero
habilita → Evals (11) · sin baseline de retrieval no se puede medir degradación
07Modelo comercial y pricing
requiere ← Observabilidad (03) · sin medir uso real, el pricing es a ciegas
requiere ← Manejo de errores (04) · cobrar SLA exige poder cumplirlo
habilita → Onboarding (08) · sin planes claros no hay self-service
11Testing y evals
requiere ← Versionado prompts (10) · sin versionado no podés comparar versiones
requiere ← Observabilidad (03) · para evals continuas en producción
habilita → Compliance (13) · evidencia auditable para regulados
13Cumplimiento normativo
requiere ← Auth (02) · trazabilidad de quién accedió a qué dato
requiere ← Observabilidad (03) · logs inmutables de auditoría
requiere ← Datos (01) · aislamiento entre tenants es legal, no técnico
14Seguridad adversarial
requiere ← Auth (02) · permisos finos para limitar agencia del modelo
requiere ← Observabilidad (03) · detectar prompt injection requiere logs
habilita → Compliance (13) · varios marcos lo exigen explícitamente

Lectura clave: Auth (02) y Observabilidad (03) son dimensiones fundacionales — bloquean el avance de varias otras. Vale la pena llevarlas a fase 02 temprano aunque parezcan "infraestructura sin valor visible".
